# 后端 AI Coding 工程评测

这是一次接近真实工作的后端工程评测，不是传统笔试。你可以使用任意 AI Coding 工具完成，包括 Claude Code、Cursor、Codex、Copilot、Trae、通义灵码等。

我们评估的不是你是否手写代码，而是你是否能在明确业务目标下，理解已有代码，使用 AI 提升效率，同时基于自己的后端认知做出正确取舍。

## 产品业务背景

本题只聚焦一个通用场景：**一对一私聊消息系统**。

可以把它理解成微信、飞书、Slack、企业 IM 里最基础的一条链路：

> 用户 A 给用户 B 发送一条文本消息。消息会出现在两个人的聊天详情页，也会影响双方最近聊天列表中的最后一条消息、最近更新时间和未读角标。

这里涉及两个页面：

1. **聊天详情页**：用户 A 和用户 B 的私聊窗口，展示具体消息列表，以及每条消息的发送状态。
2. **最近聊天列表**：类似微信首页，展示每个私聊会话的最后一条消息、最近更新时间，以及未读角标。

这里的“未读数”只指一个普通 IM 场景：

> 用户 B 没有打开和用户 A 的聊天窗口时，用户 A 发来的新消息会让用户 B 的最近聊天列表出现未读角标。用户 B 打开这个聊天窗口后，这个会话的未读数应该清零。

本题不涉及群聊、机器人、AI 助手、协作空间或复杂业务，只看一对一私聊消息链路是否可靠。

## 用户真正关心什么

用户不会关心后端用了什么状态机、索引或同步策略。用户只关心这些结果是否可信：

1. 我发出去的消息到底有没有成功。
2. 我网络不好时重复点击发送，不应该出现两条一模一样的消息。
3. 我看到消息已经发送成功，不应该过一会儿又变成发送失败。
4. 我在手机上发送成功，网页端也应该能看到同一条消息和同一个状态。
5. 对方收到新消息后，最近聊天列表里的最后一条消息和未读角标应该是对的。
6. 如果线上出现“消息重复”“成功又变失败”“另一个设备看不到”“未读数不对”，研发应该能通过日志和数据定位原因。

## 本次业务目标

`demo/` 是一个一对一私聊消息系统的早期后端原型。它可以运行，但还不适合作为真实线上能力。

产品现在希望把“发送一条私聊消息，并在多设备上可靠展示”这条链路改得更稳定。

你需要基于现有代码完成必要改造，尽量减少这些用户可见问题：

1. **消息重复**：用户因为网络抖动或重复点击，导致聊天窗口里出现多条重复消息。
2. **状态回退**：消息已经显示成功，之后又被旧结果改成失败。
3. **重试混乱**：发送失败后再次发送，导致新旧发送结果互相覆盖，或者列表里多出不该出现的消息。
4. **多设备不一致**：同一个用户在手机和网页端看到的消息状态、消息列表或最近聊天列表长期不一致。
5. **最近聊天列表错误**：最后一条消息、最近更新时间、未读角标和聊天详情页不一致。
6. **查询和排障困难**：消息变多后列表读取变慢；线上出问题时，日志无法定位到用户、设备、消息和发送尝试。

## 开发说明

真实项目里，已有代码通常不会完全干净。你可能会看到历史兼容逻辑、实验代码、旧注释或看起来可以复用的工具函数。

请不要只根据文件名、注释或 AI 的第一版判断来决定方案。你需要围绕上面的用户问题，判断哪些代码应该进入核心链路，哪些只适合保留在兼容或历史场景里。

如果 AI 在开发过程中给出多个方案，请让它说明各自的代价，再由你做最终选择。我们更希望看到有限但可靠的改造，而不是为了做题把系统大而全重写。
